SpringBoot项目中为什么使用Druid连接池？优点

Druid连接池与其他连接池相比，有什么优点？

  • 1、高效性能
    • 采用先进的池化技术，能够在高并发下，快速的准确的处理请求；
    • Druid能够自动检测和回收空闲连接，避免连接资源的浪费；有效控制连接池的大小，避免连接池中连接数过多或者过少的情况；
  • 2.可扩展性
    • Druid连接池可以通过增加数据源来处理大量的并发请求；
    • 可以动态的调整连接池的大小、超时时间等参数；
  • 3.高度的可配置性
    • Druid连接池的参数可以通过配置文件进行调整；
    • Druid还提供了一套强大的监控和诊断工具，可以帮助开发人员在应用程序中定位和解决连接池的问题；
  • 4.支持多数据源
    • 可以同时连接多个不同类型的数据库，并为每个数据源连接池的参数进行配置
  • 5.开源免费
    • 由阿里巴巴开源的
    • Druid 社区也会不断更新维护

sql优化相关

关于sql优化，你知道那些？

  • in中包含的值不应过多

    • in中的数据是存储在数组中的
    • 这个数组中的元素是排好序的
    • 如果in中的元素过多一般使用exist替代in使用

  • 区分in和exist、not in 和 not exist的使用

    • exist是以外层表作为驱动表，先被访问； exist适用于外表小内表大的情况下；
    • in是先执行子查询； in使用了内表小外表达的情况下
    • 对于not in 和 not exist情况，推荐使用 not exist，not in可能存在逻辑问题

  • select语句查询的时候务必指定字段

    • 较少不必要的资源消耗
    • 减少了使用覆盖索引（select的数据从索引中就可以获取，不必再读取数据行了）的可能性

  • 当只需要一条数据的时候，使用limit 1

    • limit是可以停止全表扫描的

  • 排序字段尽可能的添加索引

    • 索引有两种排序方式
      • 使用索引扫描：索引本来就是有序的，不必再进行排序
      • 在内存中排序：会占用CPU资源，如果查询结果太大，还会生成临时文件，在磁盘中排序，还会进行大量的IO操作，性能较差；
    • 不添加索引会占用CPU资源，如果查询结果太大，还会生成临时文件，在磁盘中排序，还会进行大量的IO操作，性能较差；

  • 限制条件的其他字段没有索引，尽量少用or

    • or两边，如果有一边没有索引，就会造成不走索引的情况；

  • 尽量使用union all代理union

    • union会将结果集合并后再进行唯一性过滤，这个过程涉及到排序操作，增加大量的CUP运算，影响性能；
    • 适应union all的前提-结果集没有重复数据；

  • 使用合理的分页方式提高分页的效率

    • limit m,n 让m尽可能的小
    • id>1000 limit 10;

  • 如果要查询的数据量过大，可以使用分段查询

  • 避免在where子句中对字段进行表达式操作

    • where语句中使用了算术运算，会造成引擎放弃索引；

  • 避免使用隐式类型转换

    • where语句中的字段类型与参数类型不一致时，会发生类型转换

  • 对于联合索引，遵守最左前缀法则

    • 创建联合索引的时候，尽可能的将常用的字段放在前面；例如 字段1，字段2，字段3

    select 字段1 走索引

    select 字段1，字段2 走索引

    select 字段2，字段3 不走索引

  • 对于联合索引，注意范围查询语句

  • 比如between、>、<等条件时，会造成后面的索引字段失效。

select * from test where a between ? and ? and b=?；a列走索引，因a列是范围查询，因此b列是无法使用索引。

什么情况下@Transaction注解会失效？

https://baijiahao.baidu.com/s?id=1661565712893820457&wfr=spider&for=pc

  • 标注在Interface上并且配置了Spring AOP 使用CGLib动态代理，将会导致@Transactional注解失效
  • @Transactional 应用在非 public 修饰的方法上
    • 在Spring AOP 代理的时候，会对目标方法进行拦截，其中有一个方法，如果不是public修饰符修饰的方法，就不会获取到@Transactional注解里属性的配置信息
  • @Transactional 注解属性 propagation 设置错误
    • 错误的设置事务的传播行为
  • @Transactional 注解属性 rollbackFor 设置错误
    • Spring默认抛出了未检查unchecked异常（继承自 RuntimeException 的异常）或者 Error才回滚事务；其他异常不会触发回滚事务。如果在事务中抛出其他类型的异常，但却期望 Spring 能够回滚事务，就需要指定 rollbackFor属性。
    • 如果发生了IO异常，是不会发生回滚的；
  • 同一个类中方法调用，导致@Transactional失效
    • 同一个类中有方法a和b,a没有使用注解，b使用了注解，调用方法a，b的事务不会生效；
    • Spring AOP代理造成的，因为只有当事务方法被当前类以外的代码调用时，才会由Spring生成的代理对象来管理。
  • 异常被你的 catch“吃了”导致@Transactional失效
    • 手动的捕获这个异常并进行处理，会认为当前事务应该正常commit。
    • 在业务方法中一般不需要catch异常，如果非要catch一定要抛出throw new RuntimeException()，或者注解中指定抛异常类型
  • 数据库引擎不支持事务
    • MySQL数据库默认使用支持事务的innodb引擎。一旦数据库引擎切换成不支持事务的myisam，那事务就从根本上失效了。

线程池的参数

https://blog.csdn.net/m0_60380027/article/details/124387652

  1. corePoolSize：核心线程数，线程池正常情况下保持的线程数，大户人家“长工”的数量。
  2. maximumPoolSize：最大线程数，当线程池繁忙时最多可以拥有的线程数，大户人家“长工”+“短工”的总数量。
  3. keepAliveTime：空闲线程存活时间，没有活之后“短工”可以生存的最大时间。
  4. TimeUnit：时间单位，配合参数 3 一起使用，用于描述参数 3 的时间单位。
  5. BlockingQueue：线程池的任务队列，用于保存线程池待执行任务的容器。
  6. ThreadFactory：线程工厂，用于创建线程池中线程的工厂方法，通过它可以设置线程的命名规则、优先级和线程类型。
  7. RejectedExecutionHandler：拒绝策略，当任务量超过线程池可以保存的最大任务数时，执行的策略。

什么时候触发策略的？

  • 拒绝策略：当线程池的任务超出线程池队列可以存储的最大值之后，执行的策略。
