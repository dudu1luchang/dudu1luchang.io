Tomcat总体架构之浏览器请求服务器流程

浏览器访问服务器的流程-》即http请求的处理过程；

http(应用层协议)：定义数据格式的；真正传输使用的TCP协议；

tcp/ip协议用于数据的传输；

三次握手需要三次连接；

三次连接发送的就是TCP连接请求；

通信的底层就是socket；

数据按照http格式进行封装（组织）；

tcp的数据是按照http格式进行组织的；

将静态数据通过浏览器呈现给用户；（静态数据：html/js/图片）

tomcat运行的时候会将jsp解析成servlet的一个类，在这个类中通过responses输出的还是html代码段；html代码段通过浏览器呈现给用户；

注意：浏览器访问服务器使⽤的是Http协议，Http是应⽤层协议，⽤于定义数据通信的格式，具体的数据传输使⽤的是TCP/IP协议

Tomcat总体架构之Tomcat处理请求大致流程

当一个请求来的时候，tomcat进行怎样的处理（也就是上一节的第五步）；

Tomcat是一个Http服务器（原因：能够接收并且处理http请求，所以tomcat是⼀个http服务器）

Tomcat不是这样做的：

如果http服务器接收到http请求的时候，调用java业务处理类；【存在问题，耦合度上升了-tomcat和业务类耦合在一起了】

tomcat不是这样做的；

Tomcat是这样做的：

http服务器将请求传递给了Servlet容器;

Servlet容器在容器中找哪个servlet能够处理；

如果Servlet发现servlet中的java业务处理类没有被加载，会将java业务处理类进行加载；

HTTP 服务器接收到请求之后把请求交给Servlet容器来处理，Servlet 容器通过Servlet接⼝调⽤业务类。Servlet接⼝和Servlet容器这⼀整套内容叫作Servlet规范。

整个Tomcat就是Servlet容器；

注意：Tomcat既按照Servlet规范的要求去实现了Servlet容器，同时它也具有HTTP服务器的功能。
Tomcat的两个重要身份
1）http服务器
2）Tomcat是⼀个Servlet容器

Tomcat总体架构之Servlet容器处理请求流程

http服务器只是与客户端完成交互；

servlet容器完成业务逻辑；

Tomcat Servlet容器处理流程

Request对象转换为ServeltRequest对象；

【定位Servlet】容器根据传进来的ServletRequest找到能够处理这个请求的Servlet(根据请求的url和servlet进行一个匹配);

【加载Servlet】底层通过反射机制完成Servlet实例化；

当⽤户请求某个URL资源时
1）HTTP服务器会把请求信息使⽤ServletRequest对象封装起来
2）进⼀步去调⽤Servlet容器中某个具体的Servlet
3）在 2）中，Servlet容器拿到请求后，根据URL和Servlet的映射关系，找到相应的Servlet
4）如果Servlet还没有被加载，就⽤反射机制创建这个Servlet，并调⽤Servlet的init⽅法来完成初始化
5）接着调⽤这个具体Servlet的service⽅法来处理请求，请求处理结果使⽤ServletResponse对象封装
6）把ServletResponse对象返回给HTTP服务器，HTTP服务器会把响应发送给客户端

Tomcat总体架构综述

对上面两点的汇总（Tomcat处理请求+Servelt处理流程）

通过上面的讲解，我们奋发tomcat有两个非常重要的功能需要完成；

1）和客户端浏览器进行交互，进行socket通信，将字节流和Request/Response等对象进行转换；

2）Servlet容器处理业务逻辑；

Tomcat设计了两个重要的组件（重要），完成特定的功能；

  • 连接器组件（Connector）：和客户端交互（socket通信）

对应角色：http服务器功能

  • 容器组件（Container）：处理业务逻辑（Servlet容器）

对应角色：servlet容器

在这两个组件的外部，还存在其他的东西，在后面我们会涉及到；

连接器组件Coyote简介及支持的协议和IO模型

Coyote 是Tomcat 中连接器的组件名称 , 是对外的接⼝。客户端通过Coyote与服务器建⽴连接、发送请求并接受响应 。

Coyote就是处理Socket连接的；

（1）Coyote 封装了底层的⽹络通信（Socket 请求及响应处理）
（2）Coyote 使Catalina 容器（容器组件）与具体的请求协议及IO操作⽅式完全解耦
（3）Coyote 将Socket 输⼊转换封装为 Request 对象，进⼀步封装后交由Catalina 容器进⾏处理，处理请求完成后, Catalina 通过Coyote 提供的Response 对象将结果写⼊输出流
（4）Coyote 负责的是具体协议（应⽤层）和IO（传输层）相关内容

对第二步的讲解：

  • http请求是应用层协议；

  • 数据在传输的时候，使用的是io模型（bio，nio等）【传输层】；

在处理请求的时候需要使用到的；

Coyote进行处理，避免了Catalina 容器进行处理；解耦了；

对第四步的讲解：

Coyote的解析处理，需要用到一个应用层，传输层；

Tomcat Coyote ⽀持的 IO模型与协议

Tomcat⽀持多种应⽤层协议和I/O模型，如下：

在使用的时候可以特定的指定某个协议的；

BIO： 阻塞io模型（同步）

NIO： 非阻塞io模型（非同步）

在 8.0 之前 ，Tomcat 默认采⽤的I/O⽅式为 BIO，之后改为 NIO。 ⽆论 NIO、NIO2 还是 APR， 在性能⽅⾯均优于以往的BIO。 如果采⽤APR， 甚⾄可以达到 Apache HTTP Server 的影响性能。

连接器组件Coyote内部组件及工作流程

详细讲解下Coyote内部组件；

里面有些许子组件：

  • EndPoint组件：处理传输层协议（TCP/IP）的；socket处理（做socket通信编码的），是对传输层的抽象；
  • Processor组件：处理应用层协议（Http）的；
  • Adapter组件：将原生的Request转换成ServletRequest;

三个组件共同完成Coyote的功能；

ProtocolHandler组件：是一个组合组件

适配器设计模式，开发中经常会使用到；
