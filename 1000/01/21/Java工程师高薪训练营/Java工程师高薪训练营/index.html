任务一：常用设计原则和设计模式

软件开发的流程

常用的设计原则；

软件 开发 是一个复杂的过程

  • 详细设计文档（可以参考类图结构）

类名、成员变量、成员方法（方法参数与返回值类型）；就只剩下方法体了；这是最初始的程序员；

详细设计文档会非常的详细

开会讨论、再修改、开会讨论、再修改、。。。。

差不多了，然后分工；

  • 编码和测试

编码过程中需要进行测试（我们进行的是一个功能测试）

开发完成后会提交给测试人员测试（代码的抗压能力，功能问题），需要不断的修改与测试

知道暂时没有大的问题

  • 安装和调试

测试也是非常消耗时间的；（不能到客户的电脑上运行就不行了）

加班很多的时候（活干不完）（客户一会让这样，一会让那样）

在整个软件的开发过程中，设计是很重要的；

经验的提升，也会到设计里面，不仅仅写代码，我们需要做的还有很多；

开闭原则和里氏代换原则

软件开发的过程中，设计非常重要；

这里讲解写常用的设计原则

开闭原则

代码写完后，尽量不要修改；对扩展开发对修改关闭；

例如：

写一个Person实体类；构造方法，getset

很多的地方用到了这个Person类，测试人员测试的没有任何的问题；然后客户改了需求，人还需要添加年龄的特征（直接在原有的Person类中加一个age属性【不建议】），这可能造成所有使用Person类的方法都要重新做；大大增加了工作量；

可以新写一个SubPerson类；继承Person类

建议、尽量不要修改；都测试完了建议对修改关闭，扩展开放；

好处：测试工作不需要做了；

里氏代换原则

任何基类可以出现的地方，子类一定可以出现

理念让我们多使用多态

屏蔽不同子类的差异性实现通用的编程，实现不一样的效果；

想打印圆形的对象，又要打印圆形的对象又要打印矩形的对象；使用基类；

建议多使用多态；

多态：父类类型的引用指向子类对象；

依赖倒转原则和接口隔离原则

依赖倒转原则

尽量去依赖抽象类，而不是具体的实现类；

原因：抽象类对子类具有强制性和规范性；

抽象类的意义：抽象类的实际意义不在于自身创建对象，而在于被继承；当一个类继承抽象类之后，必须重写所有的抽象方法，否则该类也会是抽象类；

具体实现类没有规范性和约束性；

person类中写了一个show方法，在subPerson类中没有反应；

如果将Person类变成抽象类，如果在Person中写了一个show方法，subPerson就会报错，必须要重写show方法

具体实现类管不住子类

接口隔离原则

开发中多使用小接口，而不是大接口，避免接口的污染；降低了类之间的关联；

是对依赖倒转原则的补充；

例如：写一个Animal接口

接口中写一个run()与fly()方法；

写一个Dao类，实现Animal接口，需要重写run()与fly()方法；

由于Animal接口中有fly()方法，造成了dao类可以fly了；

是对依赖倒转原则的补充；

新建两个接口RunAnimal及FlyAnimal接口；

创建接口的时候需要考虑周全，需要经验；

降低了类之间的关联；

迪米特法则和合成复用原则

迪米特法则

迪米特法则又叫做最少知道原则；

配角喜欢打听主角的事情，然后之后就死了；

一个实体类尽可能的较少其他实体之间发生相关作用；使系统功能模块相对独立；

高内聚，低耦合

类该有的都写进这类中，使用的时候能拿到用；

和其他类的功能能不关联就不要关联；

例如：在拉钩网中担任java讲师

java讲师该有的技能都尽可能拥有；

跟讲师无关的功能，尽可能的越少越好；

比如说会修水龙头，结果水龙头出问题了，就让去修水龙头；

在讲课的时候，结果敲门让去修水龙头，会不断的别打断；

这造成不断的打断自己的功能；

尽可能的高内聚，低耦合

合成复用原则

在开发中尽量多使用合成/聚合的方式，而不是继承的方式。

例如：创建了一个类B，写了一个test()方法

可以类B继承类A，可以实现（不建议）

java中只支持单继承，不能继承多继承；

造成B不能再继承B的父类了；

又想调用A中的show方法；

可以在B的类中，声明一个其他类类型的引用作为这个类的成员变量；在这个类的构造方法中，可以进行初始化；再调用其他类的方法的时候，就可以直接的调用；

在生产者消费者的案例中使用了这个方法；

这个方法会在很多的地方使用到（合成服用原则）

类中使用到其他类的方法，要想到合成服用原则；

先将这六个原则先记下来，使用的时候理解体会；

设计模式是反复使用的、被多数人知晓的、经过分类的、代码经验的总结；

懒汉式的线程同步实现

设计模式的详解（重点）

单例设计模式之前已经讲过了，分为饿汉式及懒汉式

建议使用饿汉式，懒汉式存在线程安全问题

饿汉式：【非常重要】

1）私有化构造方法，使用private关键字修饰；

2）声明本类类型的引用执行本类类型的对象并使用private static关键字修饰

3）提供公有的get方法负责将上述的对象返回出去，使用public static关键字修饰；

写的排序是2）1）3）

有多个线程同时访问这个get方法，new对象是需要时间了，还没有new完（没有将对象赋值）的时候另一个线程也进来了，也会执行new；就会创建了两个对象；这就不是单例设计模式了；

-》我们对get这个方法的调用进行进一步处理；

只能有一个线程执行new；

使用synchronized关键字，在get上加上这个关键字；只有一个线程可以进入这个方法；

也可以在方法中使用synchronized关键字

synchronized(Singleton.class){需要new的代码}

只能进入一个class

懒汉式的线程同步优化

上面可以使用单例效果，但还是不完美；

第一次调用的时候需要锁起来，第二次，第三次就不需要加锁了，需要对get中的方法优化；

只有第一次调用这个方法的时候才锁起来；

在加锁的外面加一个条件：if(null==sin)

sin 指的是我们创建的那个单例对象；

减少了流程，提高了性能；

如果在面试中写到了这样，就是完美的写法了；

注意笔试部分；

多个工厂方法模式的实现

普通工厂方法模式，有一个小问题；

调用工厂方法的时候，需要传一个字符串，如果字符串写错的话，就不会成功调用对象；会返回一个null,用null掉用方法会造成空指针异常；

使用普通工程模式可以实现对象的创建，但是传入的字符串出错会造成不会创建对象，调用发发发的时候会造成空指针异常；

解决：在工程类中写两个成员方法；

普通工厂模式是一个成员方法可以创建多个不一样的对象；

这个工厂模式是多个成员方法创建多个不一样的对象；

多个工厂方法模式-》一个方法变成了多个方法；

静态工厂方法模式的实现

多个工厂方法模式比普通工厂模式有的优点是，不用再传字符串了，避免了空指针异常的发生；

但是上面两种工厂模式，需要先new factory，；

需要先创建工厂类的对象，根据工厂类对象的引用才可以取创建对象；

-》可以直接将工厂类中的方法加上static关键字，变成静态方法；（可以看到图中工厂方法下面使用了下划线-便是静态方法）

-》直接工厂类名点调用我们的方法就就可以了；

这样我们的工厂类使用起来就更方便了；

我们什么时候使用工厂模式呢？

不管是普通，多个方法、静态工厂模式，只要是大量对象的创建且具有共同的接口时就可以使用我们的工厂设计模式；【重要】

不建议这样写；

抽象工厂模式与工厂方法模式的区别：之前是一个工厂类多个工厂方法，现在是多个工厂类，一个工厂方法；

注意：抽象工厂模式，使用的时候，还是需要new的；
